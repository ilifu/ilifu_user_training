What Are Interactive Jobs?

This demo shows an example where three dummy scripts are executed in order as part of a workflow that is run as interactive jobs.

Interactive jobs are jobs that require ongoing input or interaction from a user while they're running. They allocate compute resources and give you a live terminal session on the compute node — so you can run commands and see results in real time.

When to use interactive jobs

  • Debugging: Run scripts step by step and inspect output
  • Exploration: Test code on real hardware before submitting batch jobs
  • Interactive workflows: Scripts that need user input at runtime
  • Development: Build and test before writing the final sbatch script
Never run code on the login node! Always use sinteractive or srun to move to a compute node first.

Two Ways to Start an Interactive Session

sinteractive — Development partition

To run an interactive job on the development partition:

sinteractive

This is a wrapper for srun targeting the Devel partition. It's designed for short interactive sessions and quick allocation.

srun --pty bash — Main partition

If you want to run an interactive job on the Main partition:

srun --pty bash

You can also specify more resources:

srun --cpus-per-task=4 --mem=8G --time=1:00:00 --pty bash
For this tutorial, use sinteractive — it's the quickest way to get a development session.

The Three Scripts

There are three scripts in this tutorial directory:

  • script1_setup.sh — Set up the environment
  • script2_calculate.sh — Run a calculation (accepts an optional input parameter)
  • script3_summarise.sh — Summarise results

You can see all the scripts by listing the directory:

ls

Why this matters

Script 2 accepts an input parameter at runtime. In a batch job, you'd have to hard-code that parameter in the sbatch script before submission — no flexibility. With an interactive session, you can run script 1, inspect the output, decide what parameter to pass to script 2, and iterate until satisfied before writing the final sbatch script.

This is how interactive jobs are useful for debugging a workflow.

Running the Scripts

Step 1 — Navigate to the tutorial directory

cd session2/tutorial2
ls
# script1_setup.sh  script2_calculate.sh  script3_summarise.sh

Step 2 — Start an interactive session

sinteractive

Your prompt changes to show the compute node, e.g.: USERNAME@compute-101 $

Don't run the scripts on the login node — always start an interactive session first.

Step 3 — Run the workflow scripts

./script1_setup.sh
./script2_calculate.sh
./script3_summarise.sh

You will see that an input parameter can be given for the second script. This is why interactive scripts are useful for debugging a workflow — if this was an sbatch script you would have had to find a way to directly pass a parameter to this script before submission.

Exiting the Interactive Session

When you are done, exit the interactive session to release allocated resources back to the cluster.

exit

Alternatively, press Ctrl+D.

Both return you to the login node. Your prompt changes back to show the login node hostname.

Resource courtesy: Always exit your interactive session when done. Resources you hold are unavailable to other users.
