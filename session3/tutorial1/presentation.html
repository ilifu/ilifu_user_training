What is a computing cluster? A computing cluster is a collection of servers or nodes (those terms can be used interchangeably) that are combined to work together in a way that is more powerful than the sum of the individual parts. In this case the individual parts consist of: A login node (which is often the main interface you will be using to the cluster); the compute nodes (which are grouped together in what we call partitions, also known as queues); some permanent storage; and a fast network connecting that all together. Today I’ll be using the “Main” partition which has nodes which consist largely of 32-core/cpu servers with between 230 and 250 GiB of RAM (depending on the specific node you end up on). We have something like 8PiB of shared storage available on our Ceph cluster. And we have two different networks connecting nodes to each other; and nodes to storage. This is not a low latency network like you might find at the CHPC, but it does have a high bandwidth. What this means is that while it can transmit a large amount of data very quickly; it is not suitable for operations which require a lot of very quick communication between nodes. But I’ll talk a bit more about this later.

Parallelism

Oxford definition for parallel processing:
a mode of operation in which a process is split into parts, which are executed simultaneously on different processors attached to the same computer [or different computers attached to the same cluster].
  • A cluster includes many connected nodes, each with its own RAM & CPUs
  • A node = single computer / server / VM / machine / box
  • The work is partitioned into smaller jobs, sometimes with a partition of the dataset
Parallelism (Wikipedia)
Modern data sets in both bioinformatics and astronomy have grown so large that it’s becoming increasingly rare to be able to do meaningful research on our own PCs / laptops or even large individual servers we might find in our labs. We are usually limited by the amount of memory, CPU or storage available on these devices and that’s when we need to look at larger resources such as a computing cluster and that’s where we can really take advantage of parallelism. So what is parallelism? The Oxford definition is [read above] We have several kinds of parallelism ranging from embarrassingly parallel problems where individual parts of a computation are not dependent on others — I’m not an Astronomer so this might be the greatest example but I’m told it’s kind of representative of what might see in the image domain, but I imagine in radio astronomy if one is doing some signal processing in a volume to create a 3 dimensional image of some region in space, the individual pixels are, at least computationally, unrelated to one another. So if we’re processing this data, these can be done independently of one another. A good example of parallelism for astronomy would be to partition/split the full spectral resolution raw data (or just raw data) from a radio telescope such as MeerKat into multiple spectral windows (SPW) and process the SPWs in parallel instead of the full raw data. Each SPW is processed simultaneously and this is what the Process MeerKat pipeline does during the radio signal processing. In bioinformatics one might be doing a genome assembly and part of this might be comparing reads to an existing genome and using that as a basis for assembly. Another class of parallelism is where individual components are highly dependent on each other. Here and example from the bioinformatics world might be where we’re doing a protein folding simulation where we’re modelling individual atoms and how they’re interacting with each other. So in this kind of simulation we typically break up our larger volume into smaller volumes, but there is interaction between these smaller volumes.

Parallelism

  • Executing portions of program simultaneously
  • Possible when we have many processors (cores/CPUs)
  • Capacity dependent on structure of both hardware AND software
  • Requires overall control/coordination mechanism
    • i.e. message passing in MPI / threading / OpenMP
So compared to a serial or sequential process, a parallelism is more complicated requires multiple portions of a program to run simultaneously. This has several requirements. Firstly we need to have multiple resources available simultaneously, so this is where we have many CPU cores available. Obviously if we’re on a single core machine, we can only run one thing at a time. But, we also need the software/program to have been written in such a way that it can take advantage of these resources. In other words, we cannot simply take a piece of software and expect it to run better/faster simply by running it on a machine with more cores. The software has to be coded with parallelism in mind. This involves some overall control / coordination mechanism — so ways for individual parts of the program to communicate with each other. Typically this involves one part of the software being in charge of the overall plan — telling other bits of the software which data to analyse and combining the results at the end. This can be done in various ways but is often achieved with frameworks or libraries such as MPI (message passing interface) / traditional (old-school) threading / or OpenMP (Open Multi-processing) (which is a framework to do threading).

Parallelism on a computing cluster can look a bit different depending on the problem that you’re trying to solve. [READ SLIDE] A small note on the way our physical hardware is set up — we do not have a low-latency network like the CHPC Does with their infiniband network. When we talk about latency what we mean is the time it takes for even a small Message to get from one node to the other. The implication is that if you have the kind of problems where there Is lots of communication between nodes (like the protein folding example I mentioned earlier) then spreading The problem across multiple nodes can actually have a negative effect on your overall computation time because You end up spending a lot of time waiting for communication between nodes, rather than doing lots of computation. So if you have the kind of problem that requires that — you’re better off using say the CHPC which has an infiniband network . Our network is quite high bandwidth though, so which there might be a relatively large delay in getting A message from A to B, those messages can be big so we can move a lot of data quite quickly.

Parallelism

  • Can be achieved on a single machine / node
    • Distributes work over many CPUs
    • Typically implemented using threads / OpenMP
    • GPU
  • Or over multiple machines / nodes
    • Distributes work over many tasks, over 1+ nodes
    • Each given amount of memory to use
    • Generally requires a cluster
    • Requires a message passing interface (MPI) wrapper
      • mpirun, srun (SLURM), mpicasa (CASA 5)
      • Version of wrapper outside and inside container / venv must match
  • Hybrid parallelism? (MPI + OpenMP / MPI + GPU / …)
  • Managed on ilifu by SLURM
Typically parallelism is implemented using one of three mechanisms: It can be on a single machine or node that has multiple cores (most modern CPUs have this) — this would be usually be done using threads or OpenMP. As mentioned earlier OpenMP is a framework that makes it easier to develop multi-threaded code. Multi-threaded code works well on multi-core machines where there is shared memory because different parts of the software running in parallel can communicate directly with each other; i.e. there’s not need to go over an external network. GPUs are usually used in scenarios where you can run exactly the same bit of code on different parts of data simultaneously. This is a bit different from threading where the actual code running can be somewhat independent or doing slightly different things. When moving over to multiple machines at the same time we usually use the MPI library which makes it easy to distribute the work over many tasks. Slurm will set up the environment, by this I mean it will allocate the machines for the job and tell the job how much memory and CPU is available. MPI can then use these resources to split out the parts of the job to where they should run and combine the results together again at the end. If you are using singularity remember that a container is kind of like a standalone machine with its own operating system and software and you will need to ensure that you are using the same version of MPI inside and outside of the container.

Parallelism in SLURM

  • Implementing a normal job in SLURM
    • Will only use 1 CPU, 1 task, and 1 node
    • Default for many processes
  • Implementing a threading / OpenMP job in SLURM
    • Need to use >1 CPU, while nodes & tasks must be 1 (unless also using MPI)
    • --cpus-per-task (not inherited from #SBATCH)
    • May need to export OMP_NUM_THREADS
  • Implementing an MPI job in SLURM
    • Need to use >1 task, while nodes and CPUs can be 1
    • --nodes, --ntasks-per-node, --cpus-per-task
    • Best to wrap singularity in MPI call
    • Cannot exceed 32 CPUs (or tasks) per node
By default a job in the cluster will only have a single cpu allocated to it. This is what you’d use if you know your code has no parallelism — perhaps it’s some kind of pre-processing where everything needs to be done sequentially. Then the next level of complexity would be using threading or OpenMP. Here we will be requesting multiple cores but only on a single node. In the job scripts later we’ll see we specify a single node, a single tasks, but multiple cpus per task. Usually you need to tell the software how many threads to actually use. In the case of OpenMP this can be done by setting the OMP_NUM_THREADS environmental variable. Otherwise it can usually be done by giving it a command line parameter. This is important as some software might mistakenly assume all the cores on the machine are available for use, but this might not be the case as your job will be restricted to what it requested. Sometimes it’s not explicitly clear that your software can use threading or OpenMP, e.g. if you use numpy a lot in your calculations — there are quite a few operations in there that have been coded to use OpenMP and will take advantage of multiple cores if they’re available. Finally an MPI job requires one to use multiple tasks — these could all be on a single machine or spread across multiple machines. The number of tasks cannot exceed the number of CPUs on the machine — most of our machines have 32 cores, but there are a handful with a bit more.

SLURM – serial and multi-CPU jobs

If code is serial, i.e. doesn’t use OpenMP or MPI, increasing CPUs or nodes will not decrease execution time


#SBATCH --nodes=1
#SBATCH --ntasks-per-node=1
#SBATCH --cpus-per-task=1

python myscript.py

Using multiple CPUs within a node with OpenMP, where N is an optional number of CPUs


#SBATCH --nodes=1
#SBATCH --ntasks-per-node=1
#SBATCH --cpus-per-task=N
#SBATCH --mem-per-cpu=XGB

export OMP_NUM_THREADS=N
python myscript.py
Note: The maximum number of CPUs per node (32) will not always give the maximum speedup

SLURM – multi-task jobs

Can also specify tasks or tasks per node. Above example doesn’t require knowledge of number of node’s CPUs; below one does.


#SBATCH --ntasks=N
#SBATCH --cpus-per-task=1
#SBATCH --mem=XGB

module add openmpi
mpirun python myscript.py
                    

#SBATCH --nodes=1
#SBATCH --ntasks-per-node=N
#SBATCH --cpus-per-task=1

SLURM – multi-task and multi-CPU jobs

Using multiple nodes with MPI:


#SBATCH --nodes=N
#SBATCH --ntasks-per-node=n
#SBATCH --cpus-per-task=1
#SBATCH --mem=XGB
module add openmpi
mpirun python myscript.py

Using multiple nodes with MPI as well as multiple cores within node with OpenMP:


#SBATCH --ntasks=N
#SBATCH --cpus-per-task=n
module add openmpi
mpirun python myscript.py
Note: Need to consider that internode communication is slower than intranode communication. --mem is memory per node, so N times XGB allocated overall.

Best practices

  • Don’t run software / heavy processes / scp on the login node
    • Only submit jobs and run SLURM commands (sbatch, srun, squeue, etc)
    • Use transfer.ilifu.ac.za to transfer data (external/internal), not login node
  • Before running a large job, identify the available resources
    • Use sinfo. Don’t hog the cluster. Reduce your allocation if possible
    • Increase likelihood of jobs running with less memory and less walltime
  • Use sbatch (srun / screen / tmux / mosh are volatile)
  • Cleanup files that aren’t needed
    • Old raw data, temporary products, /scratch data, etc
  • Don’t place large files in your home directory (/users)
  • Use Singularity (you cannot install software on the nodes)
