Session 4 — Workflow Orchestration

What You'll Learn

Why Workflow Orchestration?

  • Automated dependency management between pipeline steps
  • Resume from failed steps (no reprocessing already-done work)
  • Automatic parallelization of independent tasks (scatter/gather)
  • Reproducible, portable pipelines

Nextflow

  • Groovy-based DSL: process, workflow, channels
  • SLURM integration via nextflow.config
  • A complete FASTQ processing pipeline example

WDL / Cromwell

  • Formal typed language: task, scatter, workflow
  • SLURM backend via HOCON configuration
  • An astronomical image reduction pipeline example

Prerequisites

  • Sessions 1–3 complete (especially SLURM basics from S1)
  • Java installed on the cluster (module add java)
  • Nextflow and Cromwell installed to ~/bin/
Overhead context: Workflow engines add ~15 seconds of startup overhead. They pay off when you have many samples, complex dependencies, or need resumability.
When to use direct SLURM: For single, simple jobs without dependencies, sbatch is faster and simpler. Workflows shine at scale.
