Session 3, Tutorial 1

MPI Scaling Demo — spread work across CPUs and nodes, and see why multi-node jobs can be slower.

← Session 3

What Is This Tutorial About?

Thanks to Robin Hall for creating this tutorial.

This tutorial demonstrates how to run code in parallel — spreading work across multiple CPU cores and computing nodes. You'll learn:

  • How SLURM allocates computing resources
  • How MPI (Message Passing Interface) lets different processes work together
  • How adding more CPUs usually makes code run faster, but not always

We'll use a simple Python script that adds up 10 million numbers, splitting the work across different numbers of cores and nodes. By watching how the execution time changes, you'll see parallelism in action and discover its limitations.

The Example: simple_mpi.py

This script adds all numbers from 1 to 10,000,000 in parallel. Here's how it works:

  • Rank 0 (the "leader" process) creates the full list of numbers and splits it into chunks
  • All processes receive their chunk of numbers (using Scatterv)
  • Each process independently adds up its chunk
  • Process 0 collects all the partial sums and adds them together (using Reduce)

The script also prints out which process is running and on which computer (gethostname()), the partial sum calculated by each process, and the total execution time.

This demonstrates the key MPI concepts: distributing work (Scatterv), doing work in parallel, and collecting results (Reduce).

Getting Started

Prerequisites

Create a logs directory before running any jobs:

mkdir -p logs

Setup: Using uv and OpenMPI

This project uses uv for Python package management. Dependencies are listed in pyproject.toml.

Step 1: Load OpenMPI

Check available versions with:

module avail

Load OpenMPI:

module load openmpi/5.0.3

Step 2: Create and activate a virtual environment

uv venv .venv
source .venv/bin/activate

Step 3: Install dependencies

uv sync

This installs all dependencies listed in pyproject.toml (mpi4py, numpy).

Note: When running sbatch scripts, the OpenMPI module is loaded automatically in each script (see the module load line). You only need to set up .venv once before submitting jobs.

Python Scaling Demo

Now you're ready to run the example jobs. Submit each script with sbatch, then check the results in the logs/ directory. Watch how execution time changes as you add more resources!

Demo 1: 1 CPU, 1 Node (Baseline)

What it does: Runs the script on a single CPU core.

sbatch 1_cpu_1_node.sbatch

Check the output:

cat logs/1-cpu-1-node-*.out

What to look for: Only one process (Rank 0); all work happens on one core; this is our baseline — all other runs will be compared to this time.

Demo 2: 4 CPUs, 1 Node (Good Parallelism)

What it does: Splits the work across 4 CPU cores on the same physical machine.

sbatch 4_cpus_1_node.sbatch

Check the output:

cat logs/4-cpus-1-node-*.out

What to look for: Four processes (Rank 0, 1, 2, 3), all running on the same hostname; execution time roughly 4× faster than Demo 1; each process receives ≈2.5 million numbers.

Key lesson: When processes are on the same machine, they can communicate quickly through fast shared memory — no network overhead. This is very efficient!

Demo 3: 8 CPUs, 1 Node (Maximum Single-Node Parallelism)

What it does: Uses all 8 cores on a single machine.

sbatch 8_cpus_1_node.sbatch

Check the output:

cat logs/8-cpus-1-node-*.out

What to look for: Eight processes (Rank 0 through 7), all running on the same hostname; execution time slightly faster than Demo 2 (roughly 5–8× faster than baseline); each process receives ≈1.25 million numbers.

Key lesson: Adding more cores on the same machine continues to help, but you may hit diminishing returns depending on the CPU architecture and what the cores need to share.

Demo 4: 8 CPUs, 2 Nodes (Communication Overhead)

What it does: Spreads the work across 2 different physical machines (4 cores each).

sbatch 8_cpus_2_nodes.sbatch

Check the output:

cat logs/8-cpus-2-nodes-*.out

What to look for: Eight processes split across two hostnames (4 per node); execution time will be similar to or even slower than Demo 3; the network communication between nodes is the bottleneck.

Key lesson: When processes need to communicate across physical machines, network latency becomes a major bottleneck. ILIFU doesn't have InfiniBand (a high-speed interconnect), so inter-node communication is relatively slow. Adding more nodes doesn't help — it hurts!

Understanding the Results

To quickly compare execution times across all four simple_mpi demos:

grep "total time" logs/*-*-*.out

You should see something like:

logs/1-cpu-1-node-*.out:total time: 2.345 s
logs/4-cpus-1-node-*.out:total time: 0.612 s
logs/8-cpus-1-node-*.out:total time: 0.321 s
logs/8-cpus-2-nodes-*.out:total time: 0.425 s

Compare the execution times across all four demos:

  1. Demo 1: Baseline (slowest) ~2.3 s
  2. Demo 2: Faster (~4× speedup) ~0.6 s
  3. Demo 3: Fastest (~7× speedup) ~0.3 s
  4. Demo 4: Slower than Demo 3 (network overhead) ~0.4 s

This is a real-world lesson in high-performance computing: parallelism doesn't scale infinitely. The type of interconnect and the amount of communication your code does matter greatly.

Advanced Demo: C++ with Explicit OpenMP Parallelism

For a better demonstration of how OpenMP threading helps with scaling, we provide a C++ version (hybrid_sum.cpp) with an explicit inner loop that benefits from parallelization. This makes the scaling much more visible than library-based operations like NumPy/Polars.

The Example: hybrid_sum.cpp

This C++ program sums all numbers from 1 to 100 million (10× larger than the Python example). Here's how it works:

  • Rank 0 divides the range among all processes
  • Each process sums its assigned numbers using #pragma omp parallel for
  • OpenMP parallelizes the inner loop across all available threads
  • Rank 0 collects all partial sums using MPI_Reduce

The key difference from the Python version: the inner loop is explicitly parallelizable by OpenMP, so adding threads actually helps!

Compiling the C++ Program

Before running the sbatch files, compile the program:

module load openmpi/5.0.3
mpic++ -fopenmp -O3 hybrid_sum.cpp -o hybrid_sum

This creates an executable hybrid_sum that the sbatch scripts will use.

C++ Demo 1: Single Core (Baseline)

sbatch hybrid_sum_1_cpu_1_core.sbatch
cat logs/hybrid-sum-1-core-*.out

1 MPI process, 1 OpenMP thread. Execution time is the slowest (baseline for comparison).

C++ Demo 2: 8 Cores, Pure MPI (8 Processes, 1 Thread Each)

sbatch hybrid_sum_1_node_8_cores_8mpi.sbatch
cat logs/hybrid-sum-8mpi-1thread-*.out

8 MPI processes, 1 OpenMP thread per process. Execution time roughly 8× faster than Demo 1. Pure MPI parallelism.

C++ Demo 3: 8 Cores, Hybrid (4 Processes, 2 Threads Each)

sbatch hybrid_sum_1_node_8_cores_4mpi.sbatch
cat logs/hybrid-sum-4mpi-2threads-*.out

4 MPI processes, 2 OpenMP threads per process. Execution time comparable to or slightly better than Demo 2 (also uses 8 cores total). Fewer MPI processes = less communication overhead.

Key lesson: Both Demo 2 and Demo 3 use 8 cores total, but you can achieve similar performance with fewer MPI processes by using OpenMP threading. This shows how hybrid parallelism can be more efficient than pure MPI when communication is expensive.

C++ Demo 4: 8 Cores, Maximum Threading (1 Process, 8 Threads)

sbatch hybrid_sum_1_node_8_cores_1mpi.sbatch
cat logs/hybrid-sum-1mpi-8threads-*.out

1 MPI process, 8 OpenMP threads. No inter-process communication overhead at all. With only 1 MPI process, OpenMP threads do all the parallelism internally — the most efficient approach on a single node for communication-heavy problems.

C++ Demo 5: 8 Cores Across 2 Nodes (1 Process Per Node, 4 Threads Each)

sbatch hybrid_sum_2_nodes_4_cores_1mpi.sbatch
cat logs/hybrid-sum-2nodes-1mpi-4threads-*.out

2 MPI processes (one per node), 4 OpenMP threads per process. Execution time slower than Demo 2/3/4 due to network latency between nodes. Just like in the simple_mpi examples, adding a second node hurts performance.

Comparing All C++ Results

grep "compute sum\|Total execution time" logs/hybrid-sum-*.out

Interesting patterns:

  • Single MPI process with 8 threads (0.031 s): No speedup! This suggests OpenMP threading doesn't help much for this particular loop structure, possibly due to overhead or memory contention.
  • Pure MPI scales better: 8 MPI processes (0.006 s) is faster than 4 MPI + 2 threads (0.009 s), despite using the same 8 cores. MPI is more efficient for simple loop-based parallelism than OpenMP threading in this case.
  • Multi-node penalty (0.034 s): The 2-node version is roughly 4–5× slower than single-node approaches, confirming the network latency lesson.

Comparison with Python: The C++ version shows much tighter, faster execution times (30 ms vs 750 ms baseline), but teaches the same fundamental lesson: network communication across nodes kills performance, and the choice of parallelism strategy (MPI vs threading) matters for performance.

Key Lessons

  • More cores on one node → good speedup (shared memory, no network)
  • More nodes → often slower for communication-heavy jobs without InfiniBand
  • Pure MPI beats threading for simple loop parallelism
  • Parallelism doesn't scale infinitely — measure before requesting more resources
ilifu-specific: No InfiniBand interconnect → stay on a single node for MPI jobs unless you truly need more memory than one node provides.
