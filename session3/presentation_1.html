A quick review of ilifu's resources: while ilifu is a cloud computing facility, the bulk of what we have is dedicated to the computing machines and storage we provide on our slurm computing cluster. What is a computing cluster? A computing cluster is a collection of servers or nodes (those terms can be used interchangeably) that are combined to work together in a way that is more powerful than the sum of the individual parts. In this case the individual parts consist of: A login node (which is often the main interface you will be using to the cluster); the compute nodes (which are grouped together in what we call partitions, also known as queues); some permanent storage; and a fast network connecting that all together. Today I'll be using the "Main" partition which has nodes which consist largely of 32-core/cpu servers with between 230 and 250 GiB of RAM (depending on the specific node you end up on). We have something like 8PiB of shared storage available on our Ceph cluster. And we have two different networks connecting nodes to each other; and nodes to storage. This is not a low latency network like you might find at the CHPC, but it does have a high bandwidth. What this means is that while it can transmit a large amount of data very quickly; it is not suitable for operations which require a lot of very quick communication between nodes. But I'll talk a bit more about this later. Just a quick note on the other partitions: So although the bulk of the computing power exists in the 85 compute nodes available on the Main partition, we do also have 8 High Memory nodes — six of which have around 500GiB available memory, a seventh with about 1TiB and the eighth has 1.5 TiB of ram (with 96 cores). Finally we have the GPU partition which has 7 GPU nodes (two of which are shared with Jupyter). These have various numbers and various flavours of GPUs in them.

What is Parallelism?

Oxford definition for parallel processing:
a mode of operation in which a process is split into parts, which are executed simultaneously on different processors attached to the same computer [or different computers attached to the same cluster].
  • A cluster includes many connected nodes, each with its own RAM & CPUs
  • A node = single computer / server / VM / machine / box
  • The work is partitioned into smaller jobs, sometimes with a partition of the dataset
Parallelism (Wikipedia)
Modern data sets in both bioinformatics and astronomy have grown so large that it's becoming increasingly rare to be able to do meaningful research on our own PCs / laptops or even large individual servers we might find in our labs. We are usually limited by the amount of memory, CPU or storage available on these devices and that's when we need to look at larger resources such as a computing cluster and that's where we can really take advantage of parallelism. So what is parallelism? The Oxford definition is [read above] We have several kinds of parallelism ranging from embarrassingly parallel problems where individual parts of a computation are not dependent on others — An example from radio astronomy could be where one is doing some signal processing in a volume to create a 3 dimensional image of some region in space, the individual pixels are, at least computationally, unrelated to one another. So if we're processing this data, these can be done independently of one another. Another good example of parallelism for astronomy would be to partition/split the full spectral resolution raw data (or just raw data) from a radio telescope such as MeerKat into multiple spectral windows (or SPWs) and process these SPWs in parallel instead of the full raw data. Each SPW is processed simultaneously and this is what the Process MeerKat pipeline does during the radio signal processing. In bioinformatics one might be doing a genome assembly and part of this might be aligning reads to an existing genome — these can all be done independently and is ideal for doing in parallel. Another class of parallelism is where individual components are highly dependent on each other. An example from the bioinformatics world might be where we're doing a protein folding simulation where we're modelling individual atoms and how they're interacting with each other. So in this kind of simulation we typically break up our larger volume into smaller volumes, but we need to account for the interaction of molecules on the sides of these smaller volumes.

Parallelism Requirements

  • Executing portions of program simultaneously
  • Possible when we have many processors (cores/CPUs)
  • Capacity dependent on structure of both hardware AND software
  • Requires overall control/coordination mechanism
    • i.e. message passing in MPI / threading / OpenMP
So compared to a serial or sequential process, parallelism is more complicated and requires multiple portions of a program to run simultaneously. This has several requirements. Firstly we need to have multiple resources available simultaneously, so this is where we have many CPU cores available. Obviously if we're on a single core machine, we can only run one thing at a time. But, we also need the software/program to have been written in such a way that it can take advantage of these resources. In other words, we cannot simply take a piece of software and expect it to run better/faster simply by running it on a machine with more cores. The software has to be coded with parallelism in mind. This involves some overall control / coordination mechanism — so ways for individual parts of the program to communicate with each other. Typically this involves one part of the software being in charge of the overall plan — telling other bits of the software which data to analyse and combining the results at the end. This can be done in various ways but is often achieved with frameworks or libraries such as MPI (message passing interface) / traditional (old-school) threading / or OpenMP (Open Multi-processing) (which is a framework to do threading).

Parallelism on the ilifu Cluster

  • A cluster includes many connected nodes
  • Each node has RAM and multiple cores (some nodes also have GPUs)
  • Work of job is partitioned into smaller jobs
    • Usually on independent parts of the data
Parallelism can look a bit different depending on the problem you're trying to solve and the resources available. As noted before on ilifu we have:
* many connected nodes where
* each node has many cores and a lot of RAM
* some nodes have GPUs which can add a whole extra level of parallelism
* Work is typically partitioned into smaller parts that can be processed simultaneously. This is often done by partitioning the data into smaller chunks and processing those chunks in parallel.

A small note on the way our physical hardware is set up — we do not have a low-latency network like the CHPC does with their infiniband network. When we talk about latency what we mean is the time it takes for even a small message to get from one node to the other. The implication is that if you have the kind of problems where there is lots of communication between nodes (like the protein folding example I mentioned earlier) then spreading the problem across multiple nodes can actually have a negative effect on your overall computation time because you end up spending a lot of time waiting for communication between nodes, rather than doing lots of computation. So if you have the kind of problem that requires that — you're better off using say the CHPC which has an infiniband network. Our network is quite high bandwidth though, so while there might be a relatively large delay in getting a message from A to B, those messages can be big so we can move a lot of data quite quickly.

Single-node vs Multi-node Parallelism

Single-node (shared memory)

  • Distributes work over many CPUs within one node
  • Typically implemented using threads / OpenMP
  • GPU acceleration (also single-node)
Node (32 CPUs)
Core 1 · Core 2 · … · Core 32
shared memory — fast communication

Multi-node (distributed memory)

  • Distributes work over many tasks across 1+ nodes
  • Requires MPI: mpirun, srun, mpicasa
  • MPI version inside/outside container must match
  • Hybrid parallelism: MPI + OpenMP / MPI + GPU

Tasks 1–4
⟷ network
Node B
Tasks 5–8
⚠ ilifu does not have InfiniBand. Network latency between nodes is higher than at e.g. the CHPC. Multi-node MPI adds communication overhead that can make jobs slower than single-node — the live demo will prove this.
Typically parallelism is implemented using one of three mechanisms: It can be on a single machine or node that has multiple cores (most modern CPUs have this) — this would be usually be done using threads or OpenMP. As mentioned earlier OpenMP is a framework that makes it easier to develop multi-threaded code. Multi-threaded code works well on multi-core machines where there is shared memory because different parts of the software running in parallel can communicate directly with each other; i.e. there's no need to go over an external network. GPUs are usually used in scenarios where you can run exactly the same bit of code on different parts of data simultaneously. This is a bit different from threading where the actual code running can be somewhat independent or doing slightly different things.

When moving over to multiple machines at the same time we usually use the MPI library which makes it easy to distribute the work over many tasks. Slurm will set up the environment, by this I mean it will allocate the machines for the job and tell the job how much memory and CPU is available. MPI can then use these resources to split out the parts of the job to where they should run and combine the results together again at the end. If you are using singularity remember that a container is kind of like a standalone machine with its own operating system and software and you will need to ensure that you are using the same version of MPI inside and outside of the container.

Parallelism in SLURM

Type --nodes --ntasks --cpus-per-task Notes
Serial 1 1 1 Default; no parallelism
OpenMP / Threading 1 1 N Set OMP_NUM_THREADS=N
MPI (single-node) 1 N 1 module load openmpi + mpirun
MPI (multi-node) N n×N 1 Cross-node overhead; use cautiously on ilifu
Hybrid MPI+OpenMP 1+ N n Both ntasks and cpus-per-task > 1
Tip: On ilifu, keep --nodes=1 unless your workload genuinely requires multi-node MPI. Single-node processes communicate via shared memory — no network latency.
By default a job in the cluster will only have a single cpu allocated to it. This is what you'd use if you know your code has no parallelism — perhaps it's some kind of pre-processing where everything needs to be done sequentially. Then the next level of complexity would be using threading or OpenMP. Here we will be requesting multiple cores but only on a single node. In the job scripts later we'll see we specify a single node, a single task, but multiple cpus per task. Usually you need to tell the software how many threads to actually use. In the case of OpenMP this can be done by setting the OMP_NUM_THREADS environmental variable. Otherwise it can usually be done by giving it a command line parameter. This is important as some software might mistakenly assume all the cores on the machine are available for use, but this might not be the case as your job will be restricted to what it requested. Sometimes it's not explicitly clear that your software can use threading or OpenMP, e.g. if you use numpy a lot in your calculations — there are quite a few operations in there that have been coded to use OpenMP and will take advantage of multiple cores if they're available. Finally an MPI job requires one to use multiple tasks — these could all be on a single machine or spread across multiple machines. The number of tasks cannot exceed the number of CPUs on the machine — most of our machines have 32 cores, but there are a handful with a bit more.

SLURM — Serial and Multi-CPU Jobs

Serial job (1 CPU)

#SBATCH --nodes=1
#SBATCH --ntasks-per-node=1
#SBATCH --cpus-per-task=1

python myscript.py

OpenMP / threading job (N CPUs)

#SBATCH --nodes=1
#SBATCH --ntasks-per-node=1
#SBATCH --cpus-per-task=N
#SBATCH --mem-per-cpu=XGB

export OMP_NUM_THREADS=N
python myscript.py

When to use each

  • Serial: code has no internal parallelism — increasing CPUs or nodes will not help
  • OpenMP: code uses threading internally (e.g. numpy, scikit-learn, many bioinformatics tools like BWA, STAR)

Watch out for

  • Always set OMP_NUM_THREADS explicitly — some software defaults to all cores on the node, not just your allocation
  • Maximum 32 CPUs per node on the Main partition
Note: The maximum CPUs per node (32) will not always give maximum speedup — test with fewer cores first.

SLURM — Multi-task (MPI) Jobs

Using --ntasks (SLURM allocates nodes)

#SBATCH --ntasks=N
#SBATCH --cpus-per-task=1
#SBATCH --mem=XGB

module add openmpi
mpirun python myscript.py

Using --ntasks-per-node (you control layout)

#SBATCH --nodes=1
#SBATCH --ntasks-per-node=N
#SBATCH --cpus-per-task=1
#SBATCH --mem=XGB

module add openmpi
mpirun python myscript.py

Key differences

  • --ntasks=N — SLURM decides how to spread N tasks across nodes; may use multiple nodes
  • --ntasks-per-node=N — you specify exactly how many tasks per node; requires --nodes to be set

On ilifu

  • Prefer --nodes=1 --ntasks-per-node=N to stay on one node
  • Avoids accidental multi-node allocation that adds latency
  • N cannot exceed CPUs per node (≤ 32 on Main)

SLURM — Multi-node and Hybrid Jobs

Multi-node MPI

#SBATCH --nodes=N
#SBATCH --ntasks-per-node=n
#SBATCH --cpus-per-task=1
#SBATCH --mem=XGB
module add openmpi
mpirun python myscript.py

N nodes × n tasks each = N×n total MPI processes.
--mem is per node, so N×XGB total RAM.

Hybrid MPI + OpenMP

#SBATCH --ntasks=N
#SBATCH --cpus-per-task=n
module add openmpi
mpirun python myscript.py

N MPI processes × n OpenMP threads each.
Total cores = N×n.

Remember: Inter-node communication is slower than intra-node on ilifu. Always benchmark before scaling to multiple nodes.

Best Practices

✗ Don't

  • Run software / heavy processes on the login node
  • Transfer data via the login node — use transfer.ilifu.ac.za
  • Over-allocate resources and hog the cluster
  • Use srun / screen / tmux / mosh for long jobs (volatile — job dies if connection drops)
  • Store large files in your home directory (/users)

✓ Do

  • Use sbatch for all persistent job submission
  • Use transfer.ilifu.ac.za for external/internal data transfers
  • Check available resources with sinfo before large jobs
  • Request realistic walltime and memory — shorter = higher queue priority
  • Clean up: old raw data, temp products, /scratch files
  • Use Singularity containers for reproducible software environments
