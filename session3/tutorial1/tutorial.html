Session 3, Tutorial 1 — MPI Scaling Demo

What is MPI?

MPI (Message Passing Interface) lets multiple processes run in parallel — on the same node or across different nodes — and communicate with each other by passing messages.

Key MPI concepts used in this tutorial

  • Rank: Each process gets a unique ID (rank 0, 1, 2, …)
  • Scatterv: Rank 0 distributes chunks of work to all ranks
  • Reduce: All ranks send their partial results back to rank 0 to be combined

The example task

Sum all numbers from 1 to 10,000,000. Split across N processes — each process sums its chunk, then results are combined.

Rank 0 (Leader)
Create list, Scatterv chunks, collect Reduce
Scatterv ↓   ↑ Reduce
Rank 1
Sum chunk 1
Rank 2
Sum chunk 2
Rank N
Sum chunk N

module avail
# Look for openmpi/... entries

Step 3 — Load OpenMPI

module load openmpi/5.0.3

Step 4 — Create & activate Python virtual environment

uv venv .venv
source .venv/bin/activate

uv sync

Python Demos: 1 → 4 → 8 CPUs

Demo 1 — 1 CPU, 1 Node (Baseline)

sbatch 1_cpu_1_node.sbatch
cat logs/1-cpu-1-node-*.out
  • Single process (Rank 0 only)
  • All 10M numbers summed by one core
  • Result: ~0.76 s (our baseline)

Demo 2 — 4 CPUs, 1 Node

sbatch 4_cpus_1_node.sbatch
cat logs/4-cpus-1-node-*.out
  • 4 processes on the same machine
  • Each process sums ≈2.5M numbers
  • Result: ~0.25 s (~3× speedup)

Demo 3 — 8 CPUs, 1 Node (Best single-node)

sbatch 8_cpus_1_node.sbatch
cat logs/8-cpus-1-node-*.out
  • 8 processes on the same machine
  • Each process sums ≈1.25M numbers
  • Result: ~0.155 s (~5× speedup)

Compare all three

grep "total time" logs/*-*-*.out

Demo 4 — 8 CPUs, 2 Nodes: The Surprise

sbatch 8_cpus_2_nodes.sbatch
cat logs/8-cpus-2-nodes-*.out

What to expect

  • 8 processes — 4 on node A, 4 on node B
  • Processes run on two different hostnames
  • In theory: same 8 cores, should be as fast as Demo 3

What actually happens

  • Result: ~0.173 s — SLOWER than Demo 3 (~0.155 s)!
  • Processes must communicate across the network
  • ilifu uses Gigabit Ethernet between nodes, not InfiniBand
  • Network latency exceeds any benefit from the second node

Key lesson: More nodes ≠ more speed. Network latency dominates without high-speed interconnects. Keep jobs on a single node whenever possible on ilifu.

Advanced: C++ with Hybrid MPI + OpenMP

The C++ version (hybrid_sum.cpp) sums 100 million numbers (10× larger than Python) using explicit #pragma omp parallel for inside each MPI process.

Compile first

module load openmpi/5.0.3
mpic++ -fopenmp -O3 hybrid_sum.cpp -o hybrid_sum

C++ demos available

  • 1 core, 1 MPI, 1 thread — baseline
  • 8 cores, pure MPI — 8 processes, 1 thread each
  • 8 cores, hybrid — 4 MPI × 2 threads
  • 8 cores, pure threading — 1 MPI × 8 threads
  • 2 nodes — cross-node penalty test

Key Lessons

1. Single-node scaling is good

MPI processes on the same node communicate via shared memory — very fast. Expect near-linear speedup for embarrassingly parallel work.

2. Multi-node scaling is risky on ilifu

Network latency between nodes dominates small workloads. ilifu does not have InfiniBand. The 2-node demo was slower than 8 cores on 1 node.

3. Parallelism strategy matters

For simple loops: pure MPI often beats hybrid MPI+OpenMP because fewer messages are sent. Choose based on your workload's communication pattern.

4. Diminishing returns

Going from 1→4 CPUs gave ~3× speedup. Going from 4→8 CPUs only gave ~1.7× more speedup. Some overhead is unavoidable (coordination, synchronization).

Rule of thumb: Always benchmark your specific workload. These lessons apply to this sum-of-numbers example — your scientific code may behave very differently.
