Session 3, Tutorial 2

Resource Profiling — measure how much CPU and memory your jobs actually use, and tune your requests accordingly.

← Session 3

Overview

This tutorial briefly outlines how to profile running or previously run jobs, for the purpose of making efficient use of resources on the ilifu facility. It is based on the resource allocation guide, where you can find much more information, in addition to the recordings and slides available on the ilifu website.

Why profile your jobs?

When you submit a SLURM job, you request resources upfront: CPUs, memory, and wall-time. But how much did you actually use?

  • Over-requesting memory blocks those resources for others
  • Over-requesting wall-time lowers your priority in the queue
  • Under-requesting causes your job to fail mid-run
  • Resource waste is charged to your project's allocation

The goal: request slightly more than you actually need — typically adding a 20–30% buffer on top of actual usage.

Submit Job

After cloning this repo, create the logs directory (mkdir logs). Submit initialise_array.sbatch to the SLURM queue:

sbatch initialise_array.sbatch

Store the job ID that is output when you run the job (e.g. 2605138) as an environment variable:

jobid=2605138

View Memory Usage of Running Jobs

Using sstat

Once your job is running (check with squeue -u $USER), view the maximum memory usage of the job (MaxRSS):

sstat -j $jobid -o MaxRSS

This is given in units of kB. To calculate this in GB, divide the value by 1024². This should show ~50 GB maximum memory (~53,000,000 K).

Using htop for Real-Time Monitoring

SSH onto the node where the job is running (listed with squeue -u $USER). To SSH onto the node, you must have a job running on that node, and you must have enabled authentication forwarding when you first SSH onto the cluster using the -A parameter:

# Connect with agent forwarding
ssh -A <username>@slurm.ilifu.ac.za

# Find which node your job is on:
squeue -u $USER

# SSH to that node:
ssh compute-101

# Then run htop filtered to your user:
htop -u $USER

This gives a real-time dashboard of computing resources for your different processes running on the node, and allows you to monitor how the resource usage of your job progresses in real time.

Within the first ~30 seconds of the job running, htop should show close to 100% CPU usage (all of a single core), and ~20% memory usage (corresponding to the ~50 GB maximum memory). If you weren't quick enough, run the job again and repeat this step — the node may be different. You will be logged out of the node automatically once your job has completed.

View Memory Usage of Previous Jobs

Using sacct

After the job has run, list the MaxRSS with:

sacct -j $jobid --unit=G -o JobID,JobName,MaxRSS,ReqMem

This will also list your requested memory (ReqMem) and the unit (e.g. 80 Gn = 80 GB per node, or 7.25c = 7.25 GB per core), for comparison to the used memory.

Using seff

List the efficiency of your job's use of compute resources with:

seff $jobid

This should list a moderate to high memory efficiency (≥50%, depending on how close the maximum memory usage occurred to when SLURM samples it every 20 seconds), but low CPU efficiency, which is penalised due to our wall-time being longer than necessary (but useful for allowing time for live profiling).

To view the effect of the CPU efficiency decreasing, submit the job again and watch the output of seff:

watch seff $jobid

View Wall-Time Usage

List the wall-time vs. the time allocated to the job:

sacct -j $jobid -o jobID,jobName,Elapsed,TimeLimit

Find Historical Job ID for Profiling

In order to profile your previous jobs when submitting a similar job, you may need to find the ID of a previous job to display the usage statistics. Search for a job ID within the time range that you submitted the job, using sacct:

sacct -X --name=initialise_array --starttime=2022-01-01 --endtime=2022-04-12

Accounting Groups

Each project supported on ilifu has a corresponding SLURM accounting group, against which resource usage is charged. If you are a member of multiple projects on the ilifu cluster, it is important to select the correct project accounting group when submitting a job.

List your accounting groups

sacctmgr show user $USER cluster=ilifu-slurm2021 -s format=account%25

View your default account

sacctmgr show user $USER

Change your default account

sacctmgr modify user name=${USER} set DefaultAccount=<account>

Specify account when submitting a job

Your account can be specified within the SLURM parameter --account (following #SBATCH within sbatch jobs):

#SBATCH --account=b05-pipelines-ag
