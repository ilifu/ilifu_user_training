Session 3, Tutorial 3

High-throughput Jobs — split large jobs by resource need and chain them with afterok dependencies.

← Session 3

Overview

This tutorial follows on from Tutorial 2, and is also based on the resource allocation guide. However, it shows an example of how to make jobs more efficient by using a high-throughput approach where larger jobs are split into smaller independent jobs with their own resource requirements.

You can find additional information, including video recordings of the tutorial and the slides on the ilifu website.

Scripts Overview

Assume you have a job script with multiple steps, and these steps have different resource requirements. Normally, these steps would be combined in a single script, however for this tutorial we will be saving them in separate files.

  • script_combined.py — the combined script with all steps
  • script1_setup.py — the setup stage: mainly uses CPU resources; simulates the setup stages needed for a project
  • script2_calc.py — the main calculation: mainly uses RAM memory resources, using up to 60 GB of RAM and a small amount of compute

SBatch Overview

We have also created the sbatch files for these scripts, with roughly appropriate resource requirements:

  • slurm_script1_setup.sbatch
  • slurm_script2_calc.sbatch
  • slurm_script_combined.sbatch

Submit Combined Script as a SLURM Job

After cloning this repo, first create the logs directory (mkdir logs). Submit slurm_script_combined.sbatch to the SLURM queue:

sbatch slurm_script_combined.sbatch

Store the job ID that is output when you run the combined job (e.g. 9418176) as an environment variable:

jobid_comb=9418176

View Memory and CPU Usage of Combined Job

Once your job has completed running (check with squeue -u $USER), view the Memory and CPU usage of the job using seff:

seff $jobid_comb

The output should be similar to the following:

Job ID: 9418176
Cluster: ilifu-slurm2021
User/Group: tcloete/idia-group
State: COMPLETED (exit code 0)
Cores: 1
CPU Utilized: 00:00:58
CPU Efficiency: 98.31% of 00:00:59 core-walltime
Job Wall-clock time: 00:00:59
Memory Utilized: 22.38 GB
Memory Efficiency: 27.98% of 80.00 GB

As can be seen, the CPU Efficiency is quite high, however the Memory Efficiency of this job is quite low.

Note that the above memory utilisation is based on the MaxRSS statistics which only samples the memory used by the job every 20 seconds. So there might be some variability in your results. In a more realistic scenario with longer running times, it is likely that high memory usage would be sustained longer and would also have less of an effect on variability. Alternatively, the run can be repeated multiple times to get a better estimate.

View Memory and CPU Usage of Two Scripts Separately

Next repeat the same steps for each of the two separate scripts:

sbatch slurm_script1_setup.sbatch
sbatch slurm_script2_calc.sbatch

Also remember to save their job IDs:

jobid_script1=9418178
jobid_script2=9418179

Looking at the resource efficiency of each of these scripts:

seff $jobid_script1; echo; seff $jobid_script2
Job ID: 9418178
Cluster: ilifu-slurm2021
User/Group: tcloete/idia-group
State: COMPLETED (exit code 0)
Cores: 1
CPU Utilized: 00:00:33
CPU Efficiency: 97.06% of 00:00:34 core-walltime
Job Wall-clock time: 00:00:34
Memory Utilized: 7.59 GB
Memory Efficiency: 75.90% of 10.00 GB

Job ID: 9418179
Cluster: ilifu-slurm2021
User/Group: tcloete/idia-group
State: COMPLETED (exit code 0)
Cores: 1
CPU Utilized: 00:00:27
CPU Efficiency: 100.00% of 00:00:27 core-walltime
Job Wall-clock time: 00:00:27
Memory Utilized: 55.87 GB
Memory Efficiency: 69.84% of 80.00 GB

The first script only uses 7.59 GB of RAM, while the second script uses 55.87 GB. In the combined script we in effect requested 80 GB for the whole duration of the job. However, by splitting the combined job into these two pieces we are able to only request the resources required of each piece of the job.

Result: By splitting the job, we went from 28% to 75–70% memory efficiency — without any code changes. This frees resources for other users and is better for your project's allocation.

Submit Dependent Jobs — Manual

In the previous example we submitted both scripts to Slurm at the same time to measure their resource efficiency. However, this was only possible since they only simulate a workload and don't actually create files. In a real workflow the first script's job would first have to complete before the second script could be started.

The way to do this in Slurm is by making use of job dependencies. The first setup script will be submitted normally, and the second script will be submitted with a dependency on the first script completing successfully.

sbatch slurm_script1_setup.sbatch

Remember to save the new job ID. We will assume it is saved in the jobid_script1 variable. Next submit script2, but indicate the dependency using the -d flag, and use afterok followed by a colon and the job ID number to indicate the dependency:

sbatch -d afterok:$jobid_script1 --kill-on-invalid-dep=yes slurm_script2_calc.sbatch

Dependency types

  • afterok — this job will only start once the previous job ran successfully (exit code 0)
  • afterany — this job will run irrespective of the reason for completion of the previous job
  • --kill-on-invalid-dep=yes — makes sure that this job gets cancelled if the dependency can never be satisfied (e.g., if you cancel the first job)

Submit Dependent Jobs — Automatic

We have also included a script that can submit the job dependencies without requiring any manual capturing of job IDs:

./slurm_script_dependent.sh

You should see both jobs being submitted. However, if the script doesn't run you might have to give the script execution permission by using the chmod +x command:

chmod +x slurm_script_dependent.sh
./slurm_script_dependent.sh

View Wall-Time Usage

The one resource used by all of these scripts that has not been optimised is the wall time. You can use the following command to analyse the wall-time vs. the running time of each of the jobs (remember to replace the job IDs in the command with your own):

sacct -X -j 9418176,9418178,9418179 -o jobID,jobName%20,Elapsed,TimeLimit
JobID                     JobName    Elapsed  Timelimit
------------ -------------------- ---------- ----------
9418176           combined_script   00:00:59   00:04:00
9418178             script1_setup   00:00:34   00:02:00
9418179              script2_calc   00:00:27   00:02:00

From the output it can be seen that the allocated time of these jobs was a bit too high. Most likely a total time of 1 minute 30 seconds would be more appropriate for the combined script, and 45 seconds for script1 and script2. This is based on a margin of 20–30% that was added to the running time of the jobs.
